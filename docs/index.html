🥤 Sipper

A reactive UI framework with 3 APIs. That's it.

npm install sipperjs

state(value)

Create reactive state. Returns [getter, setter]. Primitives, objects, arrays — one function for all.

control(fn)

The single lifecycle mechanism. Setup, reactive subscriptions, and cleanup — all in one.

snapshot(value)

Escape hatch. Freeze a value so it never updates. Opt out of reactivity when you need to.

Quick Start

import { h, state, control, mount } from 'sipperjs'

function Counter() {
  const [getCount, setCount] = state(0)
  const doubled = getCount() * 2  // auto-reactive

  return (
    <div>
      <p>Count: {getCount()}</p>
      <p>Doubled: {doubled}</p>
      <button onClick={() => setCount(c => c + 1)}>+</button>
    </div>
  )
}

mount(Counter, document.getElementById('app'))

That's a complete Sipper app. The component runs once. State changes update the DOM surgically — no re-renders, no VDOM diffing.

How It Works

Components run once

Unlike React, Sipper components are functions that execute once. The JSX creates a static template with reactive bindings. When state changes, only the affected DOM nodes update — no reconciliation, no diffing.

Derived values are plain functions

const doubled = getCount() * 2       // build transform wraps this
const quadrupled = doubled * 2     // cascades like a spreadsheet

A lightweight build-time transform detects variables referencing state and wraps them in arrow functions. Zero runtime cost. The developer writes plain JavaScript — the framework handles the wiring.

control() is the entire lifecycle

// Setup — runs once
control(() => {
  console.log('mounted')
  return () => console.log('unmounted')
})

// Reactive — re-runs when deps change
control(() => {
  document.title = `Count: ${getCount()}`
})

// External resources — auto-cleaned
control(() => {
  const id = setInterval(tick, 1000)
  return () => clearInterval(id)
})

No onMount. No onCleanup. No useEffect. The component function body IS setup. control() handles subscriptions and cleanup. Ownership scopes auto-dispose everything on unmount.

SSR-first

// Server
import { renderToString } from 'sipperjs/server'
const { html, state } = renderToString(App)

// Client
import { hydrate } from 'sipperjs'
hydrate(App, document.getElementById('app'))

Same components run on server and client. State is serialized, transferred, and restored — no recomputation, instant interactivity.

Event delegation

One listener per event type on the document — not per element. 1000 buttons? Still just one click listener. Faster renders, less memory, automatic cleanup.

Provide / Inject

// Parent — provides to entire subtree
provide('theme', getTheme)

// Any descendant — no prop drilling
const getTheme = inject('theme')

Design Principles

One concept > many concepts — state() replaces useState, useRef, useMemo, createSignal, ref, reactive, computed
JavaScript is the abstraction — Functions are derivations. Closures are refs. No new primitives to learn.
Components run once — No re-renders, no dependency arrays, no stale closures.
The framework prevents mistakes — Automatic cleanup, ownership scopes, immutable getters.
SSR is the default — Not an afterthought. Same components, server and client.
Names say what they do — control not effect. getCount() not count.value.

Sipper — built from first principles.
